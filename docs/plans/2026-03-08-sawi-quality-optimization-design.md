# 赛维人格分析质量优化设计

> 日期：2026-03-08
> 状态：实施中

## 背景

使用同样的提示词对比测试，我们的 agent-core 系统与 Gemini 的赛维人格分析结果存在差距：
- **内容丰富度**：Gemini 输出 ~1500 字，我们 ~500 字
- **分析深度**：Gemini 有逐行为拆解、对比排除、侧翼分析；我们只有粗粒度的分类
- **响应速度**：伪流式实现导致用户需等待全部生成完才能看到输出（速度问题本次不解决）

但我们有 Gemini 不具备的优势：**RAG 召回了真实的赛维课程文档**，而 Gemini 基于预训练知识。

## 根因分析

| 问题 | 根因 |
|------|------|
| 分析不够深入 | 赛维 Skill 的 system_prompt 只有 5 行提纲，无详细分析框架 |
| 理论覆盖不全 | top_k=10，401 个 chunk 只召回 2.5% |
| 上下文碎片化 | chunk_size=500 字符，理论被切断 |

数据层确认无问题：13 份课程文档已合并入库，401 chunks 全量覆盖。

## 方案设计

### 改动 1：升级赛维 Skill 的 system_prompt

将 system_prompt 从 5 行提纲升级为完整的 7 步分析框架：

1. 核心人格定位 — 判断号码 + 核心恐惧/欲望
2. 行为特征逐一拆解 — 每个行为对应赛维理论
3. 对比排除 — 为什么不是其他号码
4. 侧翼与变体 — 侧翼方向 + 三元组
5. 层级与状态评估 — 高阶/中阶/低阶
6. 风险预测 — 压力/下坠时的表现
7. 关系建议 — 工作 vs 私人关系策略

关键规则：
- 必须引用参考资料 [参考N]
- 区分赛维体系术语 vs 通用九型人格
- 参考资料不足时明确标注

**文件**：`kb_storage/config/skills.json`
**风险**：零（纯配置）

### 改动 2：调整 retrieval_top_k

从 10 调整为 20。

基于 context cliff 研究（~2,500 tokens 后质量下降），不盲目增大。
20 chunks × 500 chars ≈ 5,000 tokens，在可控范围内。

**文件**：`kb_storage/config/llm_settings.json`
**风险**：零（纯配置）

### 改动 3：调整 chunk_size（可选，视测试结果）

从 500 调到 800-1000 字符，overlap 从 50 调到 100-150。
需要重新切片 + 重新嵌入全部文档。

**文件**：`kb/chunking.py`
**风险**：中等（需重建向量库）

## 实施顺序

```
Step 1: 升级 system_prompt     ← 立刻见效
Step 2: 调整 top_k = 20        ← 立刻见效
Step 3: 测试                   ← 梅诗博问题对比
Step 4: 如仍不够 → chunk_size  ← 需重建向量库
```
